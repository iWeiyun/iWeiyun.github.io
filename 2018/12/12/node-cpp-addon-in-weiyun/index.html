Node.js C++ Addon在微云Electron跨平台端中的实践与应用

Posted on 2018-12-12 ｜ farren

​ 项目近期选择使用Electron来实现跨平台桌面客户端，并考虑将已经被多平台使用的WeiyunSDK、RESTfulSDK作为native扩展库引入。Node.js本身支持c++实现native addon插件，将原生c++接口封装供js调用。

​ Node.js目前有两套API可以供c++侧选择来实现v8接口的封装，NAN和N-API。NAN是受到广泛认可（包括Node官方）的三方接口层，内部兼容各种Node版本v8接口的变动，对外提供稳定的API，免去长期以来c++开发者升级node版本需要频繁兼容c++接口的苦恼；N-API则是Node官方自行推出的新接口层，并非取代原有接口，而是旨在提供更稳定更贴近c风格的API，同样是为了避免c++开发者频繁改动，更利于生态圈的发展。由于我们在启动Electron版本的计划时，Electron的稳定版本使用的8.9.x版本的Node.js，该版本的官方文档中，N-API还处于实验阶段，权衡之后，我们还是选择近来成熟稳定的NAN来为我们electron桌面端提供native封装模块。

NAN与v8

​ nan的引入也十分方便，根据nan的官方文档，只需要通过npm在工程目录下安装即可

​ Node.js c++ addon和nan的基本用法和封装示例均可以在 https://github.com/nodejs/node-addon-examples 和Node.js官方文档中找到，跟着走一遍基本就可以对整个封装的过程有个直观的了解，请务必好好看一下。简单提一些可能涉及到的概念。

Isolate

​ 代表V8引擎的一个实例。不同的Isolate实例之间状态无交集、不共享。V8引擎被初始化时，就默认创建一个Isolate。一个Isolate在任何时侯都只允许一个线程运行，它是一个独立的V8运行时或者一个独立的V8执行环境，拥有自己的堆管理器、垃圾回收器等

Handle和HandleScope

​ V8中，使用Handle类型托管JavaScript对象，Handle类型间的赋值都是直接传递对象的引用。JavaScript类型在c++中都存在对应的类，c++中使用这些类时，不能在一般的栈或堆中使用，必须托管给Handle类型，使用V8自己的垃圾回收机制来管理。

​ Handle是一个模板类，有两种类型，Local Handle和Persistent Handle。

​ Local Handle在栈上分配，其生命周期取决于HandleScope。HandleScope常在一个函数开始时被创建，而HandleScope被析构时，GC将析构该HandleScope中的全部Handle引用的对象。这里Handle的栈，虽然不属于常规意义的C++栈，但HandleScope本身只能分配在栈上，受C++作用域的限制，一般而言，当代码离开HandleScope所在的c++作用域，Handle的生命周期也将结束。因此HandleScope更像是一个Local Handle的容器，用于批量移除不需要的Local Handle。

​ Persistent Handle提供一个堆上分配的JavaScript对象的引用，只能显式移除。类似全局变量，如果需要在不止一个函数中使用某个对象，HandleScope本身受限于c++作用域无法满足要求，此时想要保持对某对象的引用，只能使用Persistent Handle。

Context

​ V8中的JavaScript代码具有自己的运行上下文，Persistent Handle的全局性也是相对于当前上下文来说的，使用时需要显示指定所在的Context。每个Context拥有自己的global proxy object，所有的Persistent Handle都是该global proxy object的属性。Context可以看作一种作用域。

分代垃圾回收Generational GC

​ 简单提一下v8的垃圾回收策略，本来觉得没必要提的，但想着之前画了些图，不贴出来的话也是浪费了当时逝去的青春…

​ v8的GC机制，基于弱分代假设，将堆进行了分代。

新生代

​ 新生代用于对象的分配，容量较小，大约只有几MB。Js中总有些根对象，如果从这些根对象开始沿着引用遍历完毕还不能访问到的对象，就需要被回收。一个v8对象最初被创建时是按顺序被分配在新生代区里的。有一个指针指向该新生代区下一个可分配区域，如果该指针到达了新生代区的末端，则开始了一次独立的清理过程，清除已死对象，为还活着的对象的生存时间计数。这种清理实际上非常频繁，对清理速度要求很高，因此v8采用了Scavenge算法，这是一种停止-复制(stop-and-copy)算法。具体做法是新生代被分为from和to两个子空间，从新生代区分配内存到触发清理都只发生在from空间内。当清理机制被触发，from空间内存活对象被复制到to空间中，然后一次性释放整个from空间。同时，to空间会检查存活对象的内存占比，如果过高，则会将经历多次回收仍存活的对象复制到老生代区中。接着，from和to子空间的指责互换。

老生代

​ 老生代容量很大，其中大部分对象长时间保持存活状态，此时再考虑stop-and-copy算法就懵圈了，回收效率将大幅下降。故老生代区主要采用标记-清除(Mark-Sweep)和标记-压缩(Mark-Compact)算法相结合的方式进行回收。大体分为两个阶段：

  • 标记阶段，从根对象开始沿引用进行深度优先搜索，对所有对象进行白灰黑三色标记。黑色表示对象本身强引用，并已经处理该对象所有子引用；灰色表示对象本身强引用，还未处理完对象中的子引用，白色为不可达的死对象。
  • 清除/压缩阶段，调整全部存活对象位置，整理内存碎片，清除死对象，释放空闲内存。

​ 尽管采用了上述方式，GC时应用程序还是需要停止等待回收完毕，当程序堆内存在大量存活对象时，仍然变得性能很差。所以实际中，v8还引入了增量标记和惰性清理机制。增量标记相比于普通标记而言，另外维护一个表记录所有黑色对象指向白色对象的指针，内存增量超过阈值就激活增量标记功能，将这类对象及时染成灰色进行扫描。惰性清理则是不猴急地清理全部死对象，可以将部分清理工作延期清理。

​ 这里只是为了青春简单地一提，若想了解内存管理具体详情，本文最后贴出了相关链接。

​ 接下来终于进入正题了，贴代码总是胜过千言万语。尽管有了官方详尽而有人性化的教程，但总是有些过于简单，实际应用中仍然可能会遇到不少问题，将其中一些比较关键的部分整理一下。

通过libuv异步回调js主线程

​ node.js是单线程的，网络及数据库异步请求的回包，需要在v8引擎主线程才能执行js回调方法。v8引擎底层依赖libuv实现了异步事件的循环和分发。libuv主要维护了一系列event loop，js代码均执行在默认主循环uv_default_loop中，native addon执行js回调都需要抛到该loop中执行，并且避免在uv_default_loop中执行耗时操作。

​ 由于网络和数据库层异步回调可能非常频繁，短时间内多次调用uv_async_send()方法，并不会获得相应次数的执行机会。查看uv_async_send()接口的实现，当handle本身的pending值还是非0值，就表明之前的发送事件并未得到处理，因此不会得到机会执行uv__async_send()这个内部接口来驱动目标loop。就因为这个机制，只能保证若有N个异步事件发送，至少能得到一次执行async_cb回调的机会。因为，我们需要自己维护一个队列，来保证每次触发时，native内所有发送的回调都能得到执行。

异步js回调与Persistent Handle

​ 解决了addon异步回调到js主线程的问题，还需要考虑的就是在执行js传递下来的的callback函数了。addon所有注册给js调用的接口，都只有一个v8::FunctionCallbackInfo\v8::Value\类型的参数，所有js实际传递的参数、返回值的引用及JS对象本身的引用都被包含在内。

​ js调用接口时传递的参数通过info[i]即可依次获取，一般进行类型检查后，即可转换成对应的Local Handle对象。前面提到过，在创建Local Handle对象前，需要声明一个HandleScope，用来管理这些Local Handle对象的生命周期。当HandleScope离开函数作用域而被销毁时，会回收所有Local Handle对象，自然包括js传下来的v8::Function类型的回调函数。这里需要用一个Persistent Handle对象来延长该闭包的生命周期，v8::Persistent\默认是NonCopyablePersistentTraits，禁用了拷贝构造，无法直接被lambda值捕获，只能传递v8::Persistent\v8::Function\的指针；同时，NonCopyablePersistentTraits析构时不会主动为内存中对象染色，所以这里还需要传递一个deleter给智能指针，在析构时调用Reset()方法，便于v8的GC适时销毁这个persistent对象。

Local Handle作为返回值

​ 我们知道Local Handle的生命周期被容器Handle Scope管理，意味着当我们需要将Local Handle作为返回值传递时，由于Handle Scope被析构，内部所有Local Handle包括当前的返回值都随时会被GC清理。

​ 但是，就如下文会提到为自定义数据结构进行封装时，不可避免地会将Local Handle作为返回值处理。这个时候，我们需要用一种EscapableHandleScope来管理这些Local Handle，这种Scope允许Local Handle“逃离”自己的管理。通过Escape()方法，将要”逃离”的参数拷贝到一个封闭的scope中，之后当前的EscapableHandleScope仍然会标记其内部所有Local Handle继而被GC清理，实际返回刚才的副本。

Mac平台OC RunLoop与uv_default_loop

​ 我们的addon中还引入了wns的framework，在mac平台下wns有相当一部分模块使用Objective-C来实现，当我们提供wnsS的Objective-C接口给js调用时，需要先用c++层包装oc接口，再进行v8层的封装。

​ 不过，由于wns的内部实现涉及通过GCD分发到Main Queue处理异步代码，libuv的事件系统并不会执行OC的MainRunLoop，从而会导致wns无法使用。更改并维护wns代码并不是一个很好的方案，我们选择定时在libuv的uv_default_loop中手动插入RunLoop，使wns中的分发到主线程的异步回调得到执行的机会。RunLoop插入的时间间隔由addon层未完成的wns请求量来动态变化。为了避免卡住node.js的主线程，目前只能选择runMode:beforeDate:的方式来启动RunLoop，确保只执行一次，以处理所有GCD分发到Main Queue的任务。
